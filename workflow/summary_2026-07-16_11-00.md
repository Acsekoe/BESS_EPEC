# Handoff Summary - 9-Bus Runtime Test and Seidel EPEC Results

Date: 2026-07-16 11:00

## Current objective

Assess whether the model can handle a larger network before implementing the nodal BESS access auction. A 9-bus Euro-style test system was created to check runtime and whether a richer topology produces a more spatially spread BESS investment pattern.

## Data and files created

New processed data files:

- `model/data/processed/market_data_9bus.json`
- `model/data/processed/ptdf_9bus.csv`

The 9-bus case keeps the same system-wide hourly demand profile and same generation portfolio as the 5-bus Euro case. Demand was **not increased**. Only spatial distribution/topology changed.

Generation placement in the 9-bus case:

- N1: wind (`RES_Wind_N1`)
- N2: base generator (`G_Base`, 30 EUR/MWh)
- N4: mid generator (`G_Mid`, 55 EUR/MWh)
- N6: PV copied from old N3 profile (`RES_PV_N6`)
- N7: CCGT (`G_CCGT`, 75 EUR/MWh)
- N8: stronger PV copied from old N5 profile (`RES_PV_N8`)
- N9: peaker (`G_Peak`, 120 EUR/MWh)

PTDF was computed using the existing `calculate_ptdf` function from `model/prepare_data.py`, with slack node N5 and line reactances in the same approximate range as the original 5-bus data.

## Runtime checks

Commands used from `model/`:

```powershell
python primal_market_clearing_model.py --data data\processed\market_data_9bus.json
python single_investor_mpec.py --data data\processed\market_data_9bus.json --no-export --max-cpu-time 120
python epec_diagonalization.py --data data\processed\market_data_9bus.json --update-rule jacobi --max-iters 1 --no-export --max-cpu-time 120
python epec_diagonalization.py --data data\processed\market_data_9bus.json --update-rule seidel --tag 9bus_seidel
```

Observed runtime:

- Primal market-clearing LP: solved optimal immediately.
- Single-investor 9-bus MPEC: optimal in about 5.8 seconds wall time.
- 2-investor Jacobi EPEC, 1 iteration: about 13.8 seconds wall time.
- 2-investor Jacobi EPEC, capped 10-iteration probe: about 142 seconds wall time.
- Full 2-investor Seidel 9-bus EPEC: converged in 17 iterations.

## Investment pattern

The 9-bus case created a richer topology, but investment is still concentrated rather than broadly spread:

| Node | Total BESS MW | Interpretation |
|---|---:|---|
| N7 | 100.00 | Fully used, valuable near the CCGT / southern congestion pattern |
| N8 | 98.06 | Strong PV node, near the zero-price midday trough |
| N2 | 11.43 | Small secondary position, effectively I2 only |
| N6 | 1.97 | Minor PV-side position |
| N1, N3, N4, N5, N9 | ~0 | Not economically attractive under this calibration |

N8 is the strongest PV-cheap node. N7 is valuable because it is electrically close enough to exploit the low-price/congestion pattern while discharging into later high-price hours.

## Price and congestion findings

Average nodal prices are close overall, but midday spreads create the investment signal:

- N8 reaches approximately 0 EUR/MWh at the strongest PV hour.
- N7 reaches approximately 18.46 EUR/MWh in the low-price window.
- Most nodes return to 75 EUR/MWh in high-price hours.

Important line loading:

- `L29` binds in early hours and is close to binding in several later hours.
- `L89` binds around the midday PV-export window, especially hours 13-15.

This congestion pattern explains why N8 and N7 dominate the BESS siting result.

## Interpretation

The 9-bus system is computationally feasible for two investors and gives more spatial structure than the 5-bus case. However, the current calibration still produces a two-main-node investment story. If the paper needs a broader spread of investments, the data should be adjusted deliberately:

- distribute PV/wind troughs across more nodes;
- reduce dominance of N8/L89 congestion;
- introduce investor heterogeneity beyond WACC;
- or proceed with the nodal access auction, which may change allocation once scarce access is priced explicitly.

## Discussion points for next work

1. Implement the nodal BESS access auction / access price `kappa[n]` as the next methodological step.
2. Consider a zonal/system-price settlement scenario where investor revenue uses system-wide lambda instead of nodal lambda. This would approximate European bidding-zone pricing and test whether uniform prices distort BESS siting and line loading.
3. After the auction mechanism is stable, scale back toward the intended four-investor thesis setup.
4. Keep reporting belief-vs-settlement gaps and max MPEC-vs-joint lambda differences in all result tables.
