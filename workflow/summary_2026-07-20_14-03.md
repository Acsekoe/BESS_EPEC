## Thread 1: optimistic dual selection

Explained the mechanism with a minimal 2-generator LP (demand exactly at one
generator's capacity -> degenerate dual, valid price interval, e.g.
[10,30] EUR/MWh depending on tie-break -- MPEC picks the top of the interval
because it's optimistic in its own objective). Connected it to a general LP
duality theorem: primal multiple-optima (several nodes/siting patterns
achieving the same system cost) forces dual degeneracy (non-unique prices),
which is the same root cause behind both the price non-uniqueness and the
observation that aggregate EPEC capacity tracks the planner's total (326.6 MW
invariant from the 2026-07-17 summary) even though siting/profit diverge.

**Changed file: `Overleaf/model_extension.tex`** -- added a
"Robustness check against a non-strategic benchmark" paragraph after the
existing "Settlement and diagnostics" paragraph (around line 260). Defines
two reportable diagnostics: the aggregate capacity gap (EPEC total vs.
planner total) and the siting divergence (L1 distance between nodal
allocation vectors at matched totals). No numeric results were added to this
file (it's formulation-only); the actual numbers live in the 2026-07-17
summary and would go in a results/synthesis table later.

## Thread 2: IEEE-9-bus dataset build-out

Started because the user wanted to replace the project's custom 9-node ring
topology with the real IEEE-9 (WSCC) network from Channi et al. 2023
("An Approach for Load Flow and Load Dispatch Analysis of IEEE-9 Bus
System"), then iterated several times to make storage/congestion
economically meaningful. All intermediate datasets below were deleted at the
end of the session -- only the final one remains.

### Iteration history (for context, files no longer exist except the last)

1. `market_data_IEEE_9Bus.json` -- real topology/PTDF from the paper's Table 2
   reactances (bus 1-9 mapped to N1-N9, slack=N1), but *only* the topology
   came from the paper; demand/RES/generators/storage economics were all
   copied unchanged from `market_data_9bus.json`. Deleted.
2. `market_data_IEEE_9Bus_full.json` -- switched to the paper's real bus
   roles: generation only at N1(slack)/N2/N3 (paper's Pmax, cost linearized
   as `2*a*Pmax+b` since the raw `b` coefficient alone is a near-zero toy
   value), load only at N5/N6/N8 (paper's peak MW, project's own diurnal
   shape), RES kept at N1/N6/N8 but scaled x0.2625 (315 MW paper-load peak vs.
   the original 1200 MW system peak) to avoid a supply/demand scale mismatch.
   Central-planner result: only ~7 MW / ~14 MWh of optimal storage system-wide
   -- too thin to be an interesting BESS study. Deleted.
3. `market_data_IEEE_9Bus_tight.json` -- removed `G_IEEE2` (cheapest thermal,
   300 MW @ 52.20 EUR/MWh) and scaled demand at N5/N6/N8 by x2.5 uniformly.
   First attempt (line limits untouched) revealed a distinct bug/artifact:
   prices pinned at curtailment level (3755-4312 EUR/MWh) at *every* node but
   N1, because G_IEEE1/G_IEEE3 can each only reach the network through one
   300 MW line (L14/L93) -- a transmission bottleneck unrelated to the
   intended scarcity, plus a genuinely degenerate-dual artifact at economically
   inactive dead-end nodes (N2/N4/N7/N9, zero local generation/demand -- same
   phenomenon as Thread 1's toy example). Fixed by scaling all 9 line limits
   x2.5 (300->750 MW). Final result: 170.72 MW / 648.77 MWh storage, LMP range
   67.15-101.24 EUR/MWh, feasible, no curtailment. Deleted (superseded by the
   congestion variant below, which subsumes it).

### Final file (kept): `model/data/processed/market_data_IEEE_9Bus_congestion.json`

Built to show **both** evening thermal scarcity **and** midday PV export
congestion simultaneously -- the tight dataset had scarcity but no
congestion (checked: max price spread across nodes was ~1e-8 at every hour,
i.e. zero, because local demand at N6/N8 always vastly exceeded local PV, so
there was never a surplus needing export regardless of line limit).

Changes from `_full.json`:
- `G_IEEE2` removed (same as `_tight`).
- Demand scaled **differently per node** instead of uniformly: N5 x4.5
  (concentrates most of the evening-scarcity-driving growth at the one load
  bus with no local RES, so it can't crowd out the PV-surplus story), N6 x0.6,
  N8 x0.6 (reduced, not increased, so local demand shrinks and PV can exceed
  it at midday).
- `RES_PV_N6`/`RES_PV_N8` capacity scaled x2.0 (on top of `_full`'s existing
  x0.2625 scaling).
- Line limits **differentiated**: `L14/L45/L93/L57/L72` (the evening-critical
  generator-side paths) stay at 750 MW; `L46/L69/L98/L78` (the 4 lines
  touching N6/N8) tightened to 120 MW so midday PV surplus there can't fully
  export.

**Verification** (`central_planner_benchmark.py --data
data/processed/market_data_IEEE_9Bus_congestion.json --node-limit-mw 100`):
solver optimal, zero curtailment, total BESS 270.05 MW / 1290.57 MWh, system
cost 551,009.41 EUR/day, LMP range 20.60-119.79 EUR/MWh. Node-hour price
matrix confirms the intended signature:
- N8 collapses at midday: 87.2 EUR/MWh normally -> 20.6-23.1 EUR/MWh at hours
  12-15, while N2/N7 stay at 119.8 EUR/MWh the same hours.
- Max price spread across all 9 nodes peaks exactly at midday (96.66-99.19
  EUR/MWh at hours 12-15) vs. ~20 EUR/MWh at night -- congestion is strongest
  precisely when PV is highest, not incidental.
- N6 shows the same pattern but weaker (73.2 -> 64.7-66.1 EUR/MWh), since its
  midday surplus (~84 MW peak) sits close to the 120 MW line limit rather
  than blowing past it like N8's (up to 269 MW peak surplus).
- N8's storage hit exactly 100 MW, its node cap -- the planner wants to
  arbitrage the congestion harder there but is capped by the (currently
  global, not per-node) `--node-limit-mw` CLI flag.

Also kept: `model/data/processed/ptdf_IEEE_9Bus_congestion.csv` (human-readable
audit export of the PTDF, not read by any model).
