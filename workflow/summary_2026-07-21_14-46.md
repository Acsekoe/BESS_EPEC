## Reproducibility check (laptop vs. home PC)

Reran the committed Seidel congestion result
(`model/output/epec/seidel_ieee9_congestion_100_seidel/`) on this machine.
First attempt (`--max-cpu-time` default 120s) did not reproduce cleanly —
investor I1's Ipopt solve repeatedly hit `Maximum CPU Time Exceeded` and the
run oscillated instead of converging. `iteration_history.csv` from the
**original** home-PC run showed the same failure mode once (iteration 4, I1),
which it recovered from — the laptop just tips the same borderline solve into
repeated timeouts. Fix: `--max-cpu-time 400` (folder tag `..._verify`).
Result: converged in 8 iterations (vs. 7 at home), aggregate capacity matched
tightly (239.0 MW / 827.0 MWh home vs. 238.9 MW / 826.9 MWh laptop), the
binding node N8's split matched to 2 decimals on both machines, and the ~11 MW
that moved between I3/I4 landed on the non-binding N3/N9 nodes — read as
economic indifference among near-price-equivalent siting options, consistent
with the caveat already in `project_context.md`, not a reproducibility
failure.

## Jacobi (`--max-iters 20`, folder `jacobi_ieee9_congestion_100_maxiter20`)

Did **not** converge — stuck in a clean period-2 limit cycle (`max_rel dP`
alternating ~0.70 / ~2.4–2.8 for all 20 iterations, never decaying), with 29
projection events (vs. 0 for Seidel), including a 208 MW → 100 MW clip at N8
on iteration 1. Root cause: under Jacobi every investor best-responds to the
same stale snapshot simultaneously, so they all rush into the same freed-up
headroom together, get jointly clipped, and repeat. Conclusion adopted: Seidel
remains the reported mechanism; the Jacobi/Seidel divergence is itself framed
as a research finding (per existing `[[epec-algorithm-preferences]]` GNEP
framing), not something to "fix" by iterating longer — the cycle showed no
sign of decay.

## Why projection isn't Jacobi-only

Worked through the mechanism in `epec_diagonalization.py`: each investor's own
MPEC solve *does* hard-respect headroom against rivals' current state
(`invest_limit = node_limit_mw - rival_power_mw`, enforced as a variable bound
in `single_investor_mpec.py`). The residual risk under Seidel comes from
**damping**: the blended update `x_new = (1-a)x_old + a*x_proposed` mixes in
the investor's own *stale* prior value, sized against an *older* rival
configuration — if rivals grew a lot earlier in the same round, that stale
component can push the joint sum slightly over the limit. This is why
`project_joint_limit()` is called unconditionally regardless of update rule,
but empirically fires only for Jacobi (0 events in both real Seidel runs this
session).

A damping=1.0 Seidel rerun was proposed to test this directly (no stale
component ever re-enters the blend) — command was given
(`--damping 1.0`, tag `ieee9_congestion_100_seidel_damping1`); **result not
yet reported back in this session.**

## Nodal access pricing — designed and implemented (untuned)

Discussed replacing the per-investor hard headroom bound + ex-post projection
with a single shared price `λ^cap_n` per node (uniform-price / "variational
equilibrium" framing for the shared-constraint GNEP), found by an outer
subgradient search rather than embedding a literal complementarity constraint
per investor (which would just relocate the private-price problem one level
down). Noted a pre-existing, unused discriminatory pay-as-bid prototype at
`model/old/auction.py` + `epec_auction_simulation.py` — different design
point, not reused here.

**Documented** in `Overleaf/model_extension.tex`: new notation row for
`λ^cap_n`, new "Planned Extension: Nodal Access Pricing" subsection (objective
change, dropped bound, complementary-slackness clearing condition). Compiled
clean with `latexmk`; build artifacts removed per the Overleaf workflow rule.

**Implemented** (`--allocation-mechanism price`, default remains
`projection` — fully backward compatible, existing committed results
unaffected):
- `model/single_investor_mpec.py`: new `capacity_price_eur_per_mw_day` param
  on `build_single_investor_mpec`. When set, drops the rival-headroom bound
  (technical cap becomes flat `node_limit_mw`), skips the
  rival-exceeds-limit validation (expected transient under tâtonnement), and
  subtracts `Σ_n λ_n·X_power[n]` in the objective via new
  `capacity_price_expr`.
- `model/epec_diagonalization.py`: new `EpecConfig` fields
  (`allocation_mechanism`, `price_step_eur_per_mw` default 0.05,
  `price_tol_mw` default 1.0), new `update_capacity_price()` subgradient step
  (`price[n] = max(0, price[n] + step * (Σx_i,n - limit))`) called instead of
  `project_joint_limit()` in price mode, `aggregate_rival_capacity()` no
  longer clamps rival totals to the limit in price mode (dispatch physics
  need the real, possibly transiently oversubscribed, capacities), and the
  convergence check now also requires per-node excess within `price_tol_mw`.
  New CLI flags: `--allocation-mechanism`, `--price-step-eur-per-mw`,
  `--price-tol-mw`.
- `model/epec_results.py`: settlement now nets each investor's
  `capacity_price_payment_eur_per_day` out of both settled-profit variants
  (previously would have been inconsistent with the MPEC objective, which
  already includes this term) — fixed before it could produce a silently
  inflated settled-profit figure. Added `capacity_price_trajectory.csv`
  export, `capacity_price_eur_per_mw` per node in `node_shares`/summary, and
  `allocation_mechanism`/price params in `run_config.json`. Fixed a related
  bug caught while editing: `print_epec_summary`'s per-node investor listing
  would have wrongly printed the new `capacity_price_eur_per_mw` key as if it
  were an investor ID.

**Verification so far:** `python -m py_compile` clean on all three files. One
3-iteration smoke test (Seidel, portfolio4, 9-bus congestion, node-limit 100,
`--price-step-eur-per-mw 0.05`, `--no-export`) ran without errors and behaved
sensibly — settlement correctly nets the capacity charge, non-congested nodes
decay toward ~0 MW as expected. **But the default price step is far too weak
for this system's scale**: after 3 iterations N8 was still ~99.8 MW
oversubscribed (started ~88–98 MW over) and the price had only reached
14.27 EUR/MW/day — nowhere near enough to meaningfully curb demand at a node
where I3/I4's generation-rent-linked marginal value is in the thousands of
EUR/day. This mirrors the damping/tolerance tuning that Seidel itself needed
early on — expected, not a bug, but real tuning work remains.
